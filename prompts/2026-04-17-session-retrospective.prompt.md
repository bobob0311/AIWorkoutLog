## 세션 프롬프트 로그
아래는 이번 세션에서 실제로 오간 요청을 다음 작업자가 빠르게 이해할 수 있도록 정리한 프롬프트 기록이다.

### 1. 하네스 재설계 요청
- 새 프로젝트를 만들고, 하네스 엔지니어링을 적용한 개발 환경을 만들고 싶다.
- `AGENT.md`를 프로젝트 헌법으로 사용하고 싶다.
- `docs/PRD`, `docs/ARCHITECTURE`, `docs/ADR` 구조를 두고 싶다.
- `.codex/commands`, `.codex/settings.json`, `scripts/execute.py`, `phases/`, `context/`를 포함한 Codex용 하네스를 만들고 싶다.

### 2. 레퍼런스 포팅 요청
- `jha0313/harness_framework`를 Codex에서 쓰기 좋게 거의 그대로 가져와 달라.
- Claude 전용 구조를 Codex 전용 구조로 치환해 달라.
- 작업 추적용 `context/` 폴더를 별도로 두고 이어받기 가능한 메모 구조를 만들고 싶다.

### 3. 문서/언어 정책 요청
- 앞으로 모든 질문과 설명을 한글로 해 달라.
- md 파일도 한글 위주로 작성해 달라.
- 작업마다 `as-is / to-be`를 남길 수 있는 JSON 구조가 있으면 좋겠다.

### 4. 제품 정의 요청
- 프로젝트는 헬스 기록 웹사이트다.
- 핵심 기능은 달력에서 날짜를 누르고 운동을 기록하는 것이다.
- Next.js, Emotion, Supabase, FSD 기준으로 설계한다.
- 부위별 볼륨을 일/주/월 기준으로 확인할 수 있어야 한다.

### 5. 로그인/제품 사용성 정교화 요청
- 로그인은 이메일이 아니라 프로필 선택 후 비밀번호 입력 정도로 단순하게 가고 싶다.
- MVP니까 UI적으로 더 고민할 수 있는 부분도 찾아 달라.
- TDD 방식으로 모든 구현을 진행하고 싶다.

### 6. 구현 진행 요청
- `step1`부터 실제로 만들어 달라.
- 이어서 `step2`, `step3`, `step4`, `step5`까지 중단하지 않으면 계속 진행해 달라.
- 외부 의존 때문에 멈춰야 하면 mock 데이터로 대체하고, 남은 작업을 기록해 달라.

### 7. 날짜 기록 모달 리팩토링 요청
- 날짜 상세 페이지 대신 달력 위 모달로 바꾸고 싶다.
- 부위에 따라 운동 버튼이 달라지고, 다른 버튼으로 즉시 바꿀 수 있어야 한다.
- 중량과 횟수에 빠른 입력 버튼이 있어야 한다.
- 총 볼륨은 `kg` 단위를 붙여 보여줘야 한다.
- 기록 저장 기능이 실제로 동작해야 한다.

### 8. UX/통계 리팩토링 요청
- 날짜 칸에서 총 볼륨과 부위 정보가 더 잘 구분되게 하고 싶다.
- 날짜 칸에 부위별 미니 차트가 보이면 좋겠다.
- 여러 월 탐색이 가능해야 한다.
- 통계에서 기간을 직접 설정할 수 있어야 한다.
- 부족 부위를 보여주되, 해석 가능한 방식이면 좋겠다.
- 운동별 랭킹 차트도 추가하고 싶다.

### 9. 세부 피드백 요청
- 날짜 칸에서 `총`은 오른쪽 아래로 내려가면 좋겠다.
- 부위 이름이 차트 앞에 보였으면 좋겠다.
- 한 글자, 두 글자, 세 글자여도 차트 시작점은 같았으면 좋겠다.

### 10. 통계 해석 개선 요청
- 부족 부위만 보여주는 건 맥락에 맞지 않는다.
- 많이 한 부위도 같이 알려주면 좋겠다.
- 부위별로 저번 기간과 비교할 수 있으면 좋겠다.
- 부족한 곳은 더 해야겠다는 생각이 들고, 많이 한 곳은 내가 거기에 집중했구나 느껴졌으면 좋겠다.

### 11. 마무리/기록 요청
- 지금까지 나눈 대화 프롬프트를 파일로 저장해 달라.
- 이번 작업에 대한 평가와 피드백도 함께 남겨 달라.

## 이번 세션의 핵심 결정
- 하네스는 `AGENT.md + .codex/commands + docs + phases + context + scripts/execute.py` 구조로 유지한다.
- `phases/{task}/index.json`이 source of truth이고, `phases/index.json`은 상위 요약 인덱스로 함께 동기화한다.
- 모든 구현은 TDD로 진행한다.
- 날짜 기록은 상세 페이지가 아니라 달력 위 모달 흐름을 메인으로 사용한다.
- 달력 셀은 총 볼륨 배지와 부위별 미니 차트를 사용한다.
- 통계는 `day/week/month/custom` 범위를 지원한다.
- 통계 해석은 `많이 한 부위 + 부족 부위 + 부위별 비교 + 운동 랭킹` 흐름으로 제공한다.
- 운동 랭킹 차트의 색상은 운동 자체가 아니라 운동이 속한 부위 색을 따른다.

## 산출물 요약
- Codex용 하네스 구조 정비
- 헬스 기록 웹사이트 MVP 구현
- 날짜 기록 모달 기반 입력 흐름 구현
- 월 이동 가능한 달력 구현
- 직접 기간 설정 가능한 통계 구현
- 많이 한 부위, 부족 부위, 부위별 비교, 운동 랭킹 차트 구현
- phase/context 문서와 상태 파일 동기화

## 평가
### 잘된 점
- 요구사항이 계속 바뀌는 상황에서도 `phases`와 `context`를 통해 방향을 잃지 않고 이어갈 수 있었다.
- TDD를 기준으로 테스트를 먼저 잠그고 구현해서 회귀를 비교적 안정적으로 막을 수 있었다.
- 단순 기능 구현이 아니라 사용자가 해석 가능한 UX로 계속 다듬어 간 흐름이 좋았다.
- 달력, 모달, 통계, 문서화가 각각 따로 놀지 않고 점점 하나의 제품 흐름으로 연결됐다.

### 아쉬운 점
- 세션 중간에 몇몇 파일에서 한글 인코딩이 깨졌고, 뒤늦게 정리하는 비용이 생겼다.
- 상위 `phases/index.json` 동기화 규칙이 문서에 늦게 반영됐다.
- 빌드 캐시 문제로 `/calendar/[date]` fallback 라우트가 한 번 오해를 일으켰다.
- 통계 인사이트는 좋아졌지만, 화면 정보 밀도가 높아져 모바일에서 추가 polish 여지가 남는다.

## 피드백
### 사용자 요청 방식에 대한 좋은 점
- 원하는 UX를 기능보다 감각과 맥락으로 설명해 줘서 방향성이 명확했다.
- “부족한 부위만이 아니라 많이 한 부위도 보여달라”처럼 해석 목표를 말해준 점이 특히 좋았다.
- 중간중간 문서화와 상태 관리 구조를 같이 챙겨 달라고 해서 장기 유지보수에 유리한 흐름이 됐다.

### 다음 세션에서 더 좋아질 점
- 새 phase를 시작할 때는 시작 전에 “이번 phase의 성공 조건”을 한두 줄로 먼저 고정하면 더 빠르다.
- UI 피드백은 지금처럼 구체적일수록 좋고, 가능하면 “왜 그렇게 느꼈는지”를 한 줄 덧붙이면 더 정확하게 맞출 수 있다.
- 외부 연동이 필요한 기능은 초반에 `mock으로 갈지 실연동으로 갈지`를 먼저 정하면 작업 속도가 더 빨라진다.
